# 변경사항 정리 (2026-03-20)

## 1) 문제 원인

기존에 7개 시나리오 중 5개가 실패했던 핵심 원인은 아래 2가지였습니다.

1. 컨테이너별 런타임 차이
- 일부 컨테이너에 `sh`, `nc`, `python`, `mkdir`, `grep` 같은 기본 유틸이 부족
- 특히 `juice_shop`은 기본 `sh`가 없음

2. 실행 경로의 환경 가정
- 실행부가 `sh`/`nc` 존재를 전제로 동작
- 병렬/재실행 시 helper 생성 및 중복 실행 제어가 약함

---

## 2) 보강 내용

### A. 시나리오 본체 보강 (`scenarios/*/scenario.sh`)

- 대상: `redis_drift` + 나머지 6개 drift 시나리오
- 공통 적용:
  - 중복 실행 방지 락 추가 (`/tmp/<scenario>/.lock`, `scenario.pid`)
  - PID 1 보호 원칙 유지
  - 실패 원인 로그 명확화
  - `BUSYBOX_BIN` fallback 경로 사용 가능

### B. 시나리오 주입 스크립트 보강 (`scenarios/scenario_setup.sh`)

- `metadata.json` 파싱을 Python 의존 없이 처리
- 컨테이너에 `sh`가 없어도 `busybox sh`로 실행 가능
- `busybox.static` 캐시 생성 경쟁 방지 락 추가
  - `scenarios/.busybox.static.lock`
- applet shim 디렉터리 구성
  - `/tmp/.scenario-bin`
  - `mkdir`, `rm`, `grep`, `sleep`, `od`, `chmod` 등 링크 구성
- 실행 시 `PATH=/tmp/.scenario-bin:$PATH` 주입
- Git Bash 경로 변환 이슈 대응
  - `docker exec` 호출에 `MSYS_NO_PATHCONV=1` 적용

### C. Backend 실행 경로 보강 (`backend/app/services/scenario_service.py`)

- backend API 실행 시에도 helper busybox 자동 준비
- `sh` 미존재 컨테이너에서 busybox shell fallback 적용
- shim 디렉터리 생성 후 PATH 주입
- 대시보드/API 경유 실행과 로컬 스크립트 실행 동작 일치

### D. 기타

- `.gitignore`에 `scenarios/.busybox.static` 추가
  - 임시 캐시 바이너리 커밋 방지

동시 동작 확인:
- 7개 모두 `running=true` 확인

중복 실행 충돌 확인:
- 동일 시나리오 재실행 시 무한 누적되지 않고 제어됨

## 5) 시나리오 외 추가 권장 사항

1. 타깃 이미지 공통 유틸 표준화
- 최소 `sh`, `nc` 포함 시 운영 안정성 상승

2. Backend 실행 응답 강화
- `scenario started`, `preflight ok`, `lock hit` 같은 상태를 응답 필드로 제공 권장

3. 대시보드 실패 원인 표준 표시
- `shell missing`, `helper prepare failed`, `lock active` 등 즉시 확인 가능하도록 노출 권장
